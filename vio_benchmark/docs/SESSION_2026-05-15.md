# Session 2026-05-15 — VIO perf debugging + recording attempt

A summary of what happened today, in plain language, with the current
state and what to do next. Written because the session got messy and
I owed you a clean status.

## Why we were debugging at all

Run `v7` (`runs/run_20260514T221547Z/`) had OpenVINS blow up scale by
12.5×. Initial guess was an algorithm bug, but the bag also showed the
front cameras publishing at 6.22 Hz instead of their configured 30 Hz —
RTF was ~0.21. Hypothesis: the estimator was being starved of frames,
not broken. Fixing the sim should fix the VIO.

## What I found, in order

1. **gz-sim was rendering on the CPU, not the GPU.** `nvidia-smi`
   showed 0 % GPU utilization while `ruby` (gz-sim's main process) burned
   200 % CPU. The 3090 sat idle with 15 MiB used. Diagnosis: ogre2 (the
   render engine) was falling back to Mesa's `llvmpipe` software
   rasterizer because GLVND inside the container had no NVIDIA EGL
   vendor ICD loaded.

2. **Root cause was a small Dockerfile + env-var gap.** GLVND needs:
   - `libglvnd0`, `libegl1`, `libgl1`, `libgles2`, `libglx0` packages
   - The env var `__EGL_VENDOR_LIBRARY_FILENAMES` pointing at the
     NVIDIA ICD JSON
   - ogre2 told to use framebuffer-object render targets, not GLX
     contexts (`OGRE_RTT_MODE=FBO`)

3. **NVIDIA GLX is a trap, not the answer.** First attempt also set
   `__GLX_VENDOR_LIBRARY_NAME=nvidia` — that crashed with
   `X Error of failed request: BadValue, Major opcode 156 (NV-GLX)`,
   because NVIDIA GLX needs a working X server with the NVIDIA driver,
   and the container has neither. EGL-only is the right path for
   headless containers.

4. **Cyclone DDS + bridge trim + stripping unused sensors.** After the
   EGL fix, cameras went 6 → 13 Hz. Three more rounds got us further:
   - Switched RMW to Cyclone DDS (SHM loopback, near-zero-copy) →
     cameras 25 Hz.
   - Removed three unused RGBD cameras (rs_left, rs_back, rs_right) from
     the model — they were rendering on the GPU every tick with no
     subscriber → cameras unchanged but freed render budget.
   - Removed the depth + point-cloud channels from `parameter_bridge`
     (no consumer needed them) → cameras 28 Hz, bridge CPU halved.

5. **GUI inside the container forces Mesa back on.** This bit me late.
   gz-sim's GUI tries to open a Qt/X11 window. Inside the container,
   X auth is broken (host's X server denies the connection), so Qt
   fails. The catch: gz-sim's GUI and its sensor cameras share one
   ogre2 render context. When the GUI is asked to render, ogre2 picks
   a context that can satisfy *both* — and that defaults to Mesa
   software when no working X+NVIDIA-GLX exists. So **GUI on → Mesa →
   slow cameras**, even with all the EGL plumbing in place.

6. **The fix to (5) is `--headless-rendering` + `gui:=false`.** Adding
   that flag to `gz sim` forces EGL offscreen render targets. Combined
   with `gui:=false` (no GUI window at all), the sensor cameras get the
   NVIDIA EGL path and run at 28 Hz with the GPU at 40 % util. That's
   the working configuration committed to the repo right now.

## The recording attempt(s)

Run `v8` (`runs/run_20260515T103115Z/`, made before the GUI fix) showed
OpenVINS scale 3.51× — much better than v7 but still wrong. Analyzing
the bag turned up two problems:

- **/rs_front_right/image was missing from the bag.** OpenVINS was
  configured for stereo, but the bag-recording command didn't subscribe
  to the right camera. Without the right cam, the stored trajectory is
  effectively mono — and **mono VIO has unobservable scale during
  forward translation** (rover driving down a tunnel). That's enough on
  its own to produce the 3.5× drift.
- **/rs_front/image was at 10 Hz, not the 28 Hz the sim was producing.**
  `ros2 bag record` with default keep-last-10 reliable QoS drops image
  frames on the subscriber's queue when the mcap writer can't keep up.
  Fix: `--max-cache-size 1 GiB` + an explicit topic list (now in
  `record_run.sh`).

So fast driving wasn't the main problem in v8. **Mono geometry +
sparse frames** was. The third recording was meant to fix both.

## What to do next

You need RViz to be able to drive. RViz alone is fine for the GPU —
it just subscribes to topics; it doesn't share a render context with
gz-sim. The only reason it was off was that *cave.launch.py* couples
gz-sim's GUI with RViz, and I disabled both together.

Two ways forward; pick one:

### Option A — RViz inside the container, X auth fixed (recommended)

On the host (one-time per login):

```bash
xhost +local:root
```

Then I (or you) start RViz in the running container:

```bash
docker compose exec sim bash -lc \
  'rviz2 -d /ws/install/explorer_r2_sim/share/explorer_r2_sim/rviz/sim.rviz'
```

That window appears on your host display. The gz-sim sim itself keeps
running headless with EGL. Cameras stay at 28 Hz.

### Option B — drive blind for 60 seconds, trust the recording

If RViz is too much hassle, you can drive without visual feedback:

```bash
docker compose exec sim ros2 run teleop_twist_keyboard teleop_twist_keyboard
```

Use `i`/`,` to go forward/back, `j`/`l` to turn, lowercase letters
for gentle inputs. The rover starts at the tunnel entrance facing
into the tunnel — `i` for a few seconds, `j` or `l` briefly to turn,
`i` again, etc. ~60 s of varied motion is enough.
